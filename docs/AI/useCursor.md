> 目前 cusor 最新版中自己的大模型自动配置了代理，导致国外的模型不能使用（比如麦克雷、gpt 等）

不少人在国内安装好 Cursor 之后会碰到一个经典问题：模型列表一直加载不出来，或者直接提示"模型加载失败"。看起来像是账号问题，其实更多时候是网络出口被墙了，Cursor 没法顺利访问 OpenAI、Anthropic 这些模型服务。

这篇文章整理了一套目前亲测可用的配置方案，只要你本地有可用代理，就可以让 Cursor 在国内顺利加载 Opus、GPT 等模型，聊天和写代码都和海外环境几乎没有区别。

## 一、问题背景：为什么会"模型加载失败"

Cursor 本质上还是一个桌面客户端，它需要自己发起网络请求去访问模型服务。如果你身处中国大陆，又没有为 Cursor 配好代理，通常会遇到下面几类现象。

- 模型下拉框里只有少数本地模型或者什么都没有
- 长时间显示"加载中"，过一会儿变成"模型加载失败"
- 某个模型列表勉强能出来，但是一调用就报网络错误

这和你在浏览器里能不能打开官网不完全一样。浏览器走的是系统或者代理工具里的规则，Cursor 作为一个单独的应用，如果不显式配置，很可能走的是被限制的直连网络。所以即使代理已经开着，Cursor 还是可能连不上模型服务。

要解决这个问题，核心思路就是：让 Cursor 的所有 HTTP 请求都经过你本地的代理工具，再由代理转发到国外模型服务。

## 二、准备工作：先确认本地代理可用

在开始配置 Cursor 之前，先确认你已经有一个能正常工作的代理工具，例如 Clash、Clash Verge、V2Ray、Surge 这一类。实际用哪款不重要，只要满足两点。

- 在浏览器中可以通过代理访问平时直接打不开的站点
- 能看到本地提供的 HTTP 代理端口

大部分图形化工具都会在界面上展示一个"本地 HTTP 端口"之类的字段。本文示例使用端口号 `7897`，如果你的工具显示的是 `7890` 或者其他数字，后面的配置把这个端口替换掉即可。

只要本地代理已经可以正常访问这些站点，剩下的工作就是告诉 Cursor 去走这个端口。

## 三、在 Cursor 中打开设置并找到 settings.json

接下来要改的是 Cursor 的配置文件 `settings.json`。你可以通过下面两种方式之一打开它。

### 方式一：通过菜单进入设置

这一种更接近日常使用习惯，适合已经习惯用设置界面的用户。

1. 点击 Cursor 左上角的"文件"菜单
2. 鼠标移动到"选项"
3. 在子菜单里选择"设置"
4. 在打开的设置界面里搜索"JSON"，或者滑到页面底部
5. 找到"打开设置（JSON）"按钮，点击之后会弹出一个编辑窗口，里面就是当前用户的 `settings.json`

这一步可以配合一张截图，截图里高亮出"文件"菜单、"选项"和"设置"三处位置，方便新用户跟着找。

### 方式二：直接从界面右上角打开

如果你经常自己改配置，更快的做法是直接打开 JSON 文件。

1. 看向 Cursor 右上角，会有一个带三个点的按钮
2. 点击之后，在弹出的菜单中找到"打开设置（JSON）"
3. 点击后同样会打开 `settings.json` 文件，后面的操作和前一种方式相同

不论你用哪一种方式，最后都要确认当前编辑的是 Cursor 的设置文件，文件名显示为 `settings.json`，内容是一个用大括号包裹的 JSON 对象。

## 四、在 settings.json 里添加代理配置

准备好之后，就可以在 `settings.json` 里加入一小段代理配置。建议先找一个位置，把光标放在最外层大括号里面，和其他键值同级，然后再粘贴下面这几行。

下面这段是一个可以直接使用的示例，如果你的代理端口是 `7897`，可以原样复制。

这一段配置告诉 Cursor 两件事：所有 HTTP 请求都走本地代理，同时关闭某些在国内网络环境下容易出问题的特性，让连接更稳定。

```json
"http.proxy": "http://127.0.0.1:7897",
"http.proxySupport": "override",
"http.noProxy": [],
"cursor.general.disableHttp2": true,
"http.proxyStrictSSL": false
```

粘贴的时候要注意两点。

- 如果前后还有其他配置项，记得在每一行之间加上逗号，保持合法的 JSON 语法
- 如果文件里已经存在类似的键，比如原来有一行 `"http.proxy": "http://127.0.0.1:7890"`，就直接把值改成新的端口，不要重复出现同名键

保存文件之后，Cursor 会尝试按新配置重建网络连接。大多数情况下不需要重启应用，如果你发现模型列表仍旧没有变化，可以关闭 Cursor 重新打开一次。

## 五、端口不是 7897 时怎么改

上面的示例默认你本地代理的 HTTP 端口是 `7897`，实际情况可能会有差异。只要知道自己代理工具提供的端口号，就把示例里的数字替换过去即可。

假设你的 Clash 或 V2Ray 显示的是 `7890`，那么对应的配置应该像这样。

```json
"http.proxy": "http://127.0.0.1:7890"
```

其他几行配置维持不变，只需要把 `http.proxy` 里端口号调整正确。改完之后再次保存 `settings.json`，给 Cursor 几秒钟时间重新连接模型服务。

如果你不确定 HTTP 端口是哪一个，可以在代理工具的设置页面里找"本地 HTTP 代理"、"端口"相关的说明，大多数图形界面都会有比较明显的标注。

## 六、如何确认代理配置已经生效

保存配置之后，可以通过几个简单的操作来判断 Cursor 有没有走上代理。

1. 打开任意项目，在侧边栏里呼出 AI 面板
2. 点击模型选择下拉框，观察是否能正常显示 OpenAI 和 Anthropic 的模型名称
3. 随便选一个模型，例如某个 GPT 或 Opus，输入一段简单的中文指令
4. 如果模型在几秒钟之内就返回了结果，说明网络已经打通

如果仍然看不到模型列表，或者每次调用都很快报错，可以从下面几个角度排查。

- 确认代理工具当前是运行状态，并且规则没有禁止 Cursor 访问外网
- 确认 `http.proxy` 里的端口号和代理工具里显示的一致
- 检查是否存在多层代理，系统层面设置与 Cursor 内部设置出现冲突

通常只要端口和工具本身没有填错，这一步就可以顺利通过。

## 七、常见问题与排查思路

在实际使用过程中，一些开发者还会遇到几类比较典型的问题。下面挑几种常见情况简要说明。

### 只改了 http.proxy，没加其他几项可以吗

在部分网络环境下，只配置 `http.proxy` 也可能工作正常。但是考虑到国内运营商和网络环境的复杂程度，为了增加成功率，建议把文中的几项配置完整保留。

- `http.proxySupport` 设置为 `override` 可以避免某些扩展或者系统代理和 Cursor 的行为打架
- `cursor.general.disableHttp2` 可以规避少数代理对 HTTP 二版本处理不佳的问题
- `http.proxyStrictSSL` 设为 `false` 可以减少自签证书或中间人证书导致的连接失败

这些选项大多是偏底层的网络细节。从使用者的角度看，只要照着示例配置即可，不必逐一去理解协议层面的差别。

### 代理开着但是模型还是很慢

代理本身的质量会直接影响模型的响应速度。如果你发现浏览器访问国外网站也很慢，Cursor 调用模型自然不会快到哪里去。

可以做一个简单测试。

- 在浏览器里打开任意国外网站或者模型官网，观察加载时长
- 如果网页本身就加载缓慢，优先考虑更换线路或者节点

在同样的网络条件下，Cursor 调用模型的延迟大致会和你访问这些网站的延迟相当，只要延迟在可接受范围内，一般不会影响日常编码体验。

### 配置了代理会不会影响访问本地和内网服务

Cursor 里的这几项设置只会作用在 Cursor 这个应用内部，不会修改系统全局的网络代理。所以它不会直接影响浏览器或者其他开发工具访问本地和内网服务。

如果你在项目里还有通过 Cursor 内置请求访问公司内网接口的需求，可以根据实际情况调整 `http.noProxy`，为内网域名添加豁免。但是大多数只用 Cursor 调模型的场景，保持默认的空数组即可。

## 八、小结

让 Cursor 在国内顺利使用 Opus 和 GPT 的关键步骤其实就两步。先确保本地有一个可用的代理工具，然后在 `settings.json` 里为 Cursor 配好 HTTP 代理端口，再加上几项兼容性设置。

配置完成之后，模型列表能够正常加载，日常写代码、重构、查文档的体验都会有明显提升。对于经常和大模型协作开发的同学来说，这几行配置是很值得花几分钟时间打通的一次性成本。
